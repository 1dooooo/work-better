# 测试有效性审计报告

> **维护说明**：当完成阶段性修复后更新本文档的对应章节和状态标记。

## 审计结论

项目有 1443 个通过的测试函数、108 个功能标记为 "✅ 已完成"。但实际运行应用后，大量功能不可用或行为不符预期。

**核心问题：测试全绿 ≠ 产品可用。** 测试体系存在结构性的有效性差距——测试在验证正确的东西（mock 行为、单元逻辑），但没有验证用户真正关心的东西（功能端到端是否可用）。

## 问题逐层分析

### 问题一：E2E 测试（L4）验证的是 mock 行为，不是真实后端

**严重程度：P0——这是导致 "测试通过但功能不可用" 的首要原因。**

#### 现状

`test/e2e/helpers.ts` 中的 `injectTauriMock()` 在浏览器里注入了完整的 `__TAURI_INTERNALS__` 假实现（约 200 行 switch-case）。所有 Playwright E2E 测试的 `invoke()` 调用都被这个 JS mock 拦截，真实 Rust 后端从未被执行。

#### 具体表现

| 测试场景 | mock 行为 | 真实行为 | 差距 |
|---------|----------|---------|------|
| F1 手动捕获 | 在 JS 数组 push 一个对象 | `Event::new()` → `EventLog::append()` → SQLite 持久化 | mock 不涉及持久化、不经过 EventLog |
| F2 飞书采集 | 返回 5 条硬编码假消息 | 调用 `FeishuMessageCollector::collect()` → 飞书 API → 事件发现任务 | mock 不经过采集器、不触发任务发现 |
| F5 调度器 | 返回 2 个假任务信息 | `Scheduler::list_tasks()` → 遍历注册任务 | mock 不反映真实注册状态 |
| 采集器管理 | 返回 1 个假采集器状态 | `CollectorManager` 查询所有注册采集器 | mock 不反映真实采集器生命周期 |

#### 影响范围

所有 20 个 E2E 测试场景（F1-F6）都受此问题影响。它们通过的原因是 mock 行为正确，而非真实系统行为正确。

#### 根因

E2E 测试最初设计为在没有真实 Tauri 后端的情况下测试前端 UI 逻辑。这在开发早期是合理的权衡，但随着后端实现完成，测试没有同步切换到真实后端。

#### 修复方向

1. 短期：为每个 E2E 场景补充一个 "真实后端" 变体测试，启动真实 Tauri app 并验证端到端行为。
2. 中期：将 E2E 测试分为两类——"前端逻辑测试"（保留 mock，验证 UI 交互）和 "集成验收测试"（真实后端，验证功能可用性）。
3. 长期：逐步将 mock E2E 测试迁移为真实后端测试，mock 仅保留用于模拟外部服务不可用的场景（如飞书 API 超时）。

---

### 问题二：前端单元测试（L1-D）验证渲染而非行为

**严重程度：P1——导致前端组件的测试覆盖率虚高，但不保证组件在真实环境中工作。**

#### 现状

`TasksView.test.tsx` 的测试用例：

```typescript
it("displays initial tasks", () => {
  render(<TasksView />);
  expect(screen.getByText("整理周报")).toBeInTheDocument();
  expect(screen.getByText("Review PR #42")).toBeInTheDocument();
});
```

这些期望基于全局 mock 返回的硬编码数据。组件实际挂载时调用 `listTasks()` 获取后端数据，但测试环境中 `invoke()` 要么静默失败，要么被 mock 为固定返回值。

#### 影响范围

16 个前端单元测试文件中的大部分存在此问题。它们验证了：
- ✅ 组件能渲染不崩溃
- ✅ 特定文本/元素存在于 DOM 中（基于 mock 数据）
- ❌ 组件能从后端获取真实数据
- ❌ 数据加载失败时的错误处理
- ❌ 数据变化后 UI 正确更新

#### 修复方向

1. 为 Vitest 配置 `@tauri-apps/api/core` 的 mock，使其返回与真实后端一致的数据结构。
2. 补充 "数据加载失败"、"空数据"、"数据更新" 等行为测试。
3. 关键组件（TasksView、EventsView、ReportsView）补充真实后端集成测试。

---

### 问题三：验收测试（L5/G）场景定义完整但 step 实现是空壳

**严重程度：P0——182 个验收场景全部不可执行，测试金字塔的顶层完全缺失。**

#### 现状

`tests/acceptance/features/` 下有 7 个 `.feature` 文件，定义了 182 个 Gherkin 场景。`tests/acceptance/src/steps/` 下有对应的 step 实现文件。

但 step 实现仅在 `AcceptanceWorld` 的 HashMap 中存字符串：

```rust
// tests/acceptance/src/steps/g1_collection.rs
#[given(regex = r"^飞书消息@提及用户$")]
fn feishu_at_mention(world: &mut AcceptanceWorld) {
    world.event_type = Some("message".into());
    world.confidence = Some(0.95);
}

#[then(regex = r"^捕获为 message 且 confidence 为 high$")]
fn assert_captured(world: &mut AcceptanceWorld) {
    // 没有实际断言——只是设置了字段
}
```

这些 step 没有：
- 调用真实的 `EventLog::append()`
- 调用 `Classifier::classify()`
- 验证事件是否真正被持久化
- 验证分类结果是否正确

#### 影响

验收测试层（测试金字塔的顶层、产品可用性的最终保障）完全不可用。architecture.md 声称的 "182 个验收场景" 实际上是 182 个场景描述，而非 182 个可执行的测试。

#### 修复方向

1. step 实现必须调用真实系统——`EventLog`、`Classifier`、`Storage`、`Scheduler` 等。
2. `AcceptanceWorld` 需要持有真实的系统实例（或指向真实系统的客户端），而非仅持有字符串字段。
3. `Then` 步骤必须有真实的断言——查询系统状态并验证结果。
4. 优先实现核心路径的 step（G1 采集 → G2 处理 → G3 存储），再扩展到其他场景。

---

### 问题四：功能索引的 "✅" 标记缺乏端到端验证

**严重程度：P1——导致进度判断失真，掩盖了真实的功能完成度。**

#### 现状

`docs/features/index.md` 将全部 108 个功能标记为 `✅ 已完成`。但这些标记的依据是：
- 代码文件存在
- 相关单元测试通过
- 组件能渲染

而非：
- 用户可以从入口操作到产出
- 数据正确持久化
- 边界情况有处理

#### 具体案例

| 功能 | 标记 | 实际情况 |
|------|------|---------|
| F1.3.1 文本输入（速记窗口） | ✅ | CaptureWindow 组件存在，但 E2E 测试只验证 mock invoke 调用，未验证事件实际进入 EventLog |
| F4.1.1 任务创建 | ✅ | TasksView 有创建表单，但单元测试验证的是硬编码数据渲染，未验证 createTask 从后端返回后 UI 更新 |
| F5.1.1 日报生成 | ✅ | 代码存在，但无端到端测试验证 "采集事件 → 处理 → 生成日报" 的完整管道 |
| F2.2.3 自动升级 | ✅ | Classifier 有升级逻辑，但无测试验证 "低置信度事件 → 实际升级到大模型 → 返回结果" |
| F3.4.6 三层一致性校验 | ✅ | 代码存在，但无测试验证 Obsidian/向量DB/结构化DB 之间的真实一致性 |

#### 修复方向

1. 引入功能完成的分级标准：
   - `⬜ 未开始`：代码不存在
   - `🟡 开发中`：代码存在，单元测试通过
   - `🧪 集成验证`：代码存在，集成测试通过（真实后端）
   - `✅ 已完成`：端到端可用，验收测试通过
2. 逐一重新评估 108 个功能的真实状态。
3. 在功能索引中增加 "验证方式" 列，记录每个功能的测试覆盖情况。

---

### 问题五：部分 E2E 测试是空壳或跳过

**严重程度：P2——进一步说明测试覆盖率数字的水分。**

#### 现状

- **F5-01 ~ F5-04**（调度器集成）：测试体只有注释，没有实际代码。

```typescript
test("F5-01: Register a scheduled task and verify it appears in the list", async ({
  page,
}) => {
  // Spec: When the TasksView is wired to the scheduler:
  // 1. On mount, call invoke("list_scheduled_tasks")
  // 2. Display real tasks from the scheduler
  // ...（只有注释，没有测试代码）
});
```

- **F3-01 ~ F3-04**（处理管线）：标记为 skip，原因是 "处理管线 UI 未实现"。

architecture.md 声称 "8 场景通过，4 场景 skipped"，但实际上通过的 8 个场景中有 4 个是测试硬编码数据渲染（F5-*-integration），而非测试真实功能。

#### 修复方向

1. 删除或标记空壳测试，避免给人 "已测试" 的错觉。
2. skip 的测试应关联到具体的 issue/任务，有明确的解除条件。

## 问题全景

```
用户视角：                "我启动了 app，很多功能不能用"
                              ↑
功能索引：                108 个 ✅（基于代码存在 + 单元测试通过）
                              ↑
测试结果：                1443 通过 / 0 失败
                              ↑
测试内容：
  L1 单元测试 ✅          验证函数逻辑正确（纯计算，不涉及系统集成）
  L2 集成测试 ✅          验证 crate 间数据流（Rust 内部集成，不涉及前端）
  L3 契约测试 ✅          验证 API 形状（请求/响应格式，不涉及业务逻辑）
  L4 E2E 测试 ⚠️          验证前端行为（但用 mock 拦截了所有后端调用）
  L5 验收测试 ❌          场景已定义，step 实现是空壳
```

问题的本质：**每一层测试都通过了，但没有一层测试验证了 "用户输入 → 系统处理 → 用户可见输出" 这个完整链条。**

## 修复路线图

### 阶段一：建立真实后端测试基础（优先级 P0）

- [x] 创建真实后端测试基础设施——启动 Tauri app 或直接调用 Rust 命令的测试 harness
- [x] 为 5 个核心功能各写 1 条真实后端 happy-path 测试：
  - 手动捕获 → 事件出现在事件列表
  - 创建任务 → 任务出现在任务列表
  - 采集器开关 → 采集器状态正确变化
  - 调度器暂停/恢复 → 状态正确切换
  - 事件标记处理 → 状态正确更新
- [ ] 修复验收测试 step 实现，至少覆盖 G1（信息采集）的 37 个场景

### 阶段二：重新评估功能完成度（优先级 P1）

- [x] 按新的完成标准重新评估 108 个功能
- [x] 更新 `docs/features/index.md` 中的状态标记
- [x] 建立功能完成度看板，区分 "代码存在" 和 "端到端可用"

### 阶段三：补全测试覆盖（优先级 P2）

- [ ] 将 mock E2E 测试逐步迁移为真实后端测试
- [ ] 补充前端单元测试的行为验证（数据加载、错误处理、状态更新）
- [ ] 实现完整的 182 个验收场景 step
- [ ] 删除或标记空壳测试

### 阶段四：建立防退化机制（优先级 P2）

- [ ] CI 中增加真实后端测试门禁
- [ ] 功能索引的 ✅ 标记需要关联通过的验收测试编号
- [ ] 新功能必须有至少一条端到端测试才能标记为 ✅
