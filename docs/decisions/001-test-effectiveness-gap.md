## 背景

项目积累了 1443 个通过的测试函数（覆盖 L1-L4 + H 层），功能索引标记 108 个功能全部 "✅ 已完成"。然而实际启动 Tauri 应用后，大量功能不可用或行为不符预期。

核心矛盾：**测试全绿 ≠ 产品可用**。这不是个别的测试遗漏，而是测试体系在架构层面存在系统性的有效性差距。

## 决策

承认当前测试体系存在结构性缺陷，将 "修复测试有效性差距" 作为专项任务处理。具体决策：

1. **重新定义功能完成标准**——"✅" 必须意味着 "从用户视角端到端可用"，而非 "代码存在且单元测试通过"。
2. **在 L2（集成）和 L4（E2E）之间引入一层 "真实后端集成测试"**——不 mock Tauri IPC，直接调用真实 Rust 命令。
3. **验收测试（L5/G 层）的 step 实现必须调用真实系统**，禁止在 HashMap 里存字符串充数。
4. **E2E 测试逐步从纯 mock 过渡到真实后端**，至少覆盖每个功能的一条 happy path。

## 理由

详细的问题分析见 [测试有效性审计报告](../testing/test-effectiveness-audit.md)。以下是关键发现的摘要：

### E2E 测试（L4）测的是 mock 行为

`test/e2e/helpers.ts` 通过 `injectTauriMock()` 注入完整 `__TAURI_INTERNALS__` 假实现，所有 `invoke()` 被 JS mock 拦截，真实 Rust 命令从未执行。

### 前端单元测试（L1-D）验证渲染而非行为

组件测试基于 mock 数据验证 DOM 内容，不验证组件能否从后端获取真实数据。

### 验收测试（L5/G）step 实现是空壳

182 个 cucumber 场景的 step 实现仅在 HashMap 中存字符串，不调用真实系统。

### AI 辅助开发的结构性倾向

agent 倾向于写 "让测试通过" 的代码。当测试用 mock 绕过真实逻辑时，agent 可以让所有测试通过而不需要让任何真实功能运转。

## 替代方案

### 方案 A：只补测试，不改架构

在现有层级上补充更多测试用例。

**放弃原因**：问题不在于测试数量不足，而在于测试方向错误。增加更多 mock 测试只会加剧问题。

### 方案 B：全部改用真实后端 E2E 测试

废弃所有 mock 测试，统一用 Playwright + 真实 Tauri app 测试。

**放弃原因**：mock 测试在验证前端 UI 逻辑时仍有价值。真实后端测试更慢。应该分层共存。

### 方案 C（采用）：分层修复 + 引入真实后端层

保留现有 L1-L3 测试的价值，重点修复 L4 和 L5 的结构性缺陷，新增一层真实后端集成测试。

## 后果

### 正面

- 功能完成状态将反映真实可用性
- AI agent 的后续开发将被真实测试约束
- 回归测试能捕获前后端集成问题

### 负面

- 真实后端测试比 mock 测试更慢
- 需要维护测试基础设施
- 已有测试中的一部分需要重写
- 功能索引需要逐一重新验证，短期内会暴露大量未完成功能
