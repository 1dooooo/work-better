# 功能完成标准

> **维护说明**：当标准变更时更新本文档。

## 背景

根据 [测试有效性审计报告](../testing/test-effectiveness-audit.md)，原有功能索引的 ✅ 标记缺乏端到端验证，导致进度判断失真。本文档定义新的功能完成标准，用于重新评估 108 个功能的真实状态。

## 完成标准分级

### ⬜ 未开始
- 代码不存在
- 或仅有接口定义，无实现

### 🟡 开发中
- 代码存在
- 单元测试通过（L1）
- 但未经过集成测试或端到端验证

### 🧪 集成验证
- 代码存在
- 单元测试通过（L1）
- 集成测试通过（L2）—— 真实后端验证
- 但未经过端到端用户流程验证

### ✅ 已完成
- 代码存在
- 单元测试通过（L1）
- 集成测试通过（L2）
- 端到端测试通过（L4/L5）—— 真实后端 + 真实 UI
- 用户可以从入口操作到产出
- 数据正确持久化
- 边界情况有处理

## 评估方法

### 第一步：检查代码存在性

```bash
# 查找相关代码文件
find crates src -name "*.rs" -o -name "*.ts" -o -name "*.tsx" | xargs grep -l "功能关键词"
```

### 第二步：检查测试覆盖

```bash
# 查找相关测试
find crates src -name "*test*" -o -name "*spec*" | xargs grep -l "功能关键词"
```

### 第三步：验证测试类型

1. **单元测试（L1）**：测试单个函数或组件
2. **集成测试（L2）**：测试模块间交互，使用真实后端
3. **契约测试（L3）**：测试 API 格式
4. **E2E 测试（L4）**：测试完整用户流程
5. **验收测试（L5）**：测试业务场景

### 第四步：判定完成等级

根据测试覆盖情况，判定功能的完成等级：

| 测试覆盖 | 完成等级 |
|---------|---------|
| 无测试 | ⬜ 未开始 |
| 仅 L1 | 🟡 开发中 |
| L1 + L2 | 🧪 集成验证 |
| L1 + L2 + L4/L5 | ✅ 已完成 |

## 特殊情况

### 外部依赖功能

对于依赖外部服务的功能（如飞书采集）：
- Mock 外部服务的测试可视为 L2
- 真实外部服务的测试可视为 L4

### UI 组件功能

对于 UI 组件功能：
- 组件渲染测试可视为 L1
- 组件交互测试可视为 L2
- 完整用户流程测试可视为 L4

### 配置功能

对于配置管理功能：
- 配置读写测试可视为 L2
- 配置生效测试可视为 L4
