# 处理层架构

> **维护说明**：当处理路径、模型策略、审核规则或 SLA 配置变更时更新本文档。
> 处理层的输出格式变更需同步更新 [事件模型](event-model.md)。

## 概述

处理层是系统的智能核心。它消费 EventLog 中的事件，进行分类、提取、分析，输出结构化的 WorkRecord 写入存储层。

## 架构

```
EventLog
    │
    ▼
┌──────────────────────────────────┐
│         Classifier（分类器）       │
│  规则引擎 + 小模型，判断事件类型    │
│  输出：处理路由                    │
└──────────┬───────────────────────┘
           │
     ┌─────┴─────┬──────────┬──────────────┐
     ▼           ▼          ▼              ▼
 即时处理     聚合处理     模式分析       直接归档
     │           │          │              │
     └─────┬─────┴──────────┴──────┬───────┘
           ▼                       ▼
    ModelRouter               (直接写入)
     小模型 → 大模型
           │
           ▼
    ReviewAgent（审核代理）
           │
     ┌─────┼─────────┐
     ▼     ▼         ▼
  approved needs_fix needs_review
     │     │         │
     ▼     ▼         ▼
  写入存储  返回重处理  推送用户
```

## Classifier（分类器）

事件进入处理层的第一站，决定事件走哪条处理路径。

### 分类依据

- 事件类型（EventType）
- 来源置信度（Confidence）
- 内容特征（是否包含任务关键词、是否是回复/决策）

### 路由规则

```
即时处理：
  - 任务状态变更（task_update）
  - 审批决策（approval）
  - 会议结束（meeting，含待办）
  - @提及的消息（message，含@）
  - 用户手动捕获（manual_note）
  - SLA: P0/P1 优先级

聚合处理：
  - 普通消息（message）
  - 文档变更（document_change）
  - 浏览器访问（browsing）
  - 应用活动（app_activity）
  - SLA: P2 优先级

模式分析：
  - 需要长周期数据的任务
  - 效率趋势、工作模式、目标进展
  - SLA: P3 优先级

直接归档：
  - 低置信度事件（confidence = low）
  - 纯归档类事件
  - 不触发智能处理
```

## ModelRouter（模型路由器）

管理大小模型的调用策略和升级机制。

### 模型定义

```
模型类型        用途                    成本    速度
──────────────────────────────────────────────────
小模型          分类、标签、实体识别、    低      快
                简单摘要（< 500 字）
大模型          深度摘要、跨事件关联、    高      慢
                模式识别、报告生成
```

### 升级阈值

按任务类型分别设定，可配置：

```
任务类型              小模型默认阈值    说明
──────────────────────────────────────────────
实体提取（人/项目）    0.7              小模型对此较弱
任务识别              0.6              小模型较好
摘要生成              0.6              长度 > 500 字也触发升级
情感/态度判断          0.8              小模型较弱
关联分析              0.7              小模型较弱
模式识别              直接用大模型      小模型不胜任
```

### 升级流程

```
小模型处理
    │
    ├── confidence ≥ 阈值 → 输出结果，进入审核
    │
    └── confidence < 阈值 → 升级为大模型
            │
            ├── 大模型处理成功 → 输出结果，进入审核
            │
            └── 大模型也失败 → 标记为"待人工处理"
                  进入用户通知队列
```

### Token 预算管理

```
TokenBudget {
  daily_limit: number        // 每日 token 上限
  daily_used: number         // 已使用
  daily_remaining: number    // 剩余

  // 预算不足时的策略
  overload_strategy: 'queue_tomorrow' | 'degrade_to_small' | 'allow_overflow'
}
```

- 默认策略：非紧急任务排队等待次日，紧急任务允许突破预算但通知用户
- 预算使用情况纳入审计记录

## SLA 管理

### 优先级定义

```
优先级    适用场景                          SLA         超时行为
──────────────────────────────────────────────────────────────
P0        审批待处理、截止日今天              5 分钟      强制升级大模型 + 推送通知
P1        会议结束提取待办、@提及的任务       30 分钟     升级大模型
P2        文档变更摘要、消息聚合              4 小时      正常流程
P3        行为模式、趋势分析                 每日批量     排队处理
```

### 超时扫描

- 每 5 分钟扫描一次处理队列
- 超时事件自动提升优先级
- 每日生成 SLA 达标报告

## ReviewAgent（审核代理）

处理层输出在写入存储层之前的质量门禁。

### 审核分层

```
审核层        成本    能力
──────────────────────────────────────────
规则层        零      格式校验、必填字段、状态流转合法性
小模型层      低      一致性检查、关键信息覆盖度
大模型层      高      复杂摘要的语义审核（按需）
```

### 审核策略（按输出类型）

```
输出类型              审核方式
──────────────────────────────────────────
直接归档类            不审核
低置信度提取          规则层审核
高置信度提取          抽样审核（10% 概率）
任务状态变更          规则层 + 若涉及共享数据则推送用户
报告/摘要             规则层 + 小模型层
涉及他人的信息        规则层 + 小模型层 + 推送用户
```

### 审核结果

```
ReviewResult {
  verdict: ReviewVerdict  // approved | needs_fix | needs_review
  issues: Issue[]         // 发现的问题列表
  reviewer: string        // 审核层标识（rule | small_model | large_model）
  confidence: number      // 审核置信度
}

Issue {
  type: string            // 问题类型
  severity: string        // 严重程度
  description: string     // 问题描述
  suggestion: string      // 修复建议
}
```

### 反馈循环

- 审核问题记录到审计日志
- 同类问题频繁出现 → 调整 prompt 或升级阈值
- 用户修正记录 → 用于优化后续处理

## 全链路审计

### 审计记录

每个处理步骤产生一条 ProcessingAudit 记录（详见 [事件模型](event-model.md)）。

### 审计链路追踪

同一事件的所有审计记录通过 `trace_id` 串联，形成完整的处理链路：

```
[事件] → classifier → extract(小) → upgrade → extract(大) → review(rule) → review(小) → persist
  │         │            │           │           │             │              │           │
  └─audit───┴──audit─────┴──audit───┴──audit───┴──audit───────┴──audit───────┴──audit───┘
                              trace_id: xxx-xxx-xxx
```

### 可查询维度

| 查询目标 | 查询方式 |
|---------|---------|
| 小模型在哪类任务上经常升级 | 按 task_type 统计 upgrade 比例 |
| 大模型升级后置信度提升幅度 | 对比同一 trace_id 中小/大模型 confidence |
| 哪个 prompt 模板效果最好 | 按 prompt_id 统计 confidence 和 review 通过率 |
| 用户最常修正哪类错误 | 按 step 统计 user_correction 频率 |
| 处理成本分布 | 按 model + step 统计 token 和成本 |
| SLA 达标率 | 统计 deadline vs 实际完成时间 |
