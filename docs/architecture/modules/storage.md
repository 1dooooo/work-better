# 存储层架构

> **维护说明**：当存储结构、目录组织、保鲜策略或同步机制变更时更新本文档。
> 存储层的数据结构变更需同步更新 [事件模型](event-model.md)。

## 概述

存储层是所有数据的最终归宿。它接收处理层输出的 WorkRecord 和系统产生的审计数据，负责持久化、索引、组织。

## 三层存储架构

```
存储层
  │
  ├── Layer 1: Obsidian Vault（文档层——人的视图）
  │     ├── 以 markdown 文件为载体
  │     ├── 支持双向链接、标签、图谱
  │     ├── 用户可以直接浏览和编辑
  │     └── 是数据的"最终真相源"
  │
  ├── Layer 2: 向量数据库（RAG 层——语义检索）
  │     ├── 对文档内容做 embedding
  │     ├── 支持语义搜索
  │     └── 是查询加速层，不持有独占数据
  │
  └── Layer 3: 结构化数据库（数据层——机器的视图）
        ├── SQLite 本地数据库
        ├── 存储任务状态、审计记录、事件索引、配置
        ├── 支持精确查询和聚合统计
        └── 是程序逻辑的运行时依赖
```

### 层间关系

```
                    写入
处理层输出 ──────────────▶ Obsidian Vault
                              │
                              │ 同步
                              ▼
                         向量数据库（embedding）
                              │
                              │ 索引
                              ▼
                         结构化 DB（元数据）

                    读取
呈现层 ◀──────────────── 三层联合查询
```

### 关键原则

- **Obsidian 是唯一真相源**——向量 DB 和结构化 DB 都是从 Obsidian 派生的索引，丢失后可重建
- **双向同步**——用户在 Obsidian 中手动编辑内容后，索引层自动更新
- **写入顺序**：先写 Obsidian → 再同步向量 DB → 最后更新结构化 DB

### 信息的多维归属

同一条信息可以同时出现在多个位置：

```
例如：2026-06-06 的周一站会记录

出现在：
  日记/2026-06-06.md          ← 时间维度
  项目/项目A/会议/...md       ← 项目维度
  人/张三.md                  ← 人的维度

标签：#会议 #项目A #待办提取 #周一
双向链接：[[项目A]] [[张三]] [[用户调研]]
```

### Obsidian 原生能力利用

- **双向链接** `[[xxx]]`：自动建立信息关联网络
- **标签** `#xxx`：多维度分类，支持 dataview 查询
- **dataview 插件**：动态聚合视图
- **图谱视图**：可视化信息关联
- **模板**：不同类型内容使用不同模板

## 向量数据库

### 存储内容

- 文档 embedding（标题 + 正文）
- 事件 embedding（摘要）
- 原始文本片段（用于 RAG 召回）

### 更新策略

- 新文档写入 → 自动 embedding
- 文档修改 → 重新 embedding（延迟 5 分钟，避免频繁更新）
- 文档删除 → 移除 embedding

### 查询能力

- 语义搜索："上周讨论的性能优化方案"
- 相似文档："和这个文档相关的内容"
- 上下文召回：给大模型提供相关历史作为上下文

## 结构化数据库（SQLite）

### 表结构

```
events              # 原始事件索引（指向 Obsidian 中的文件）
work_records        # WorkRecord 索引
tasks               # 任务状态（快速查询、状态流转）
processing_audits   # 审计记录
collectors          # 采集器状态和配置
models              # 模型配置和使用统计
sync_log            # 三层同步状态（防止数据不一致）
```

## 数据一致性

### 写入流程

```
1. 写入 Obsidian markdown 文件
2. 写入成功后，触发向量 DB embedding（异步）
3. 更新结构化 DB 元数据
4. 记录 sync_log
```

### 一致性检查（定期执行）

- Obsidian 文件 ↔ 结构化 DB 记录是否匹配
- 向量 DB 的 embedding 数量 ↔ Obsidian 文档数量
- 不一致记录标记为"待修复"，自动或手动触发重建索引

## 信息保鲜引擎

### 信息腐坏类型

```
腐坏类型        示例                              检测方式
──────────────────────────────────────────────────────────
状态过期        任务已完成但标记仍为"进行中"        与飞书数据源对比
内容过期        文档已更新但摘要还是旧版本          与源文档变更时间对比
关联断裂        双向链接指向已删除/移动的文件        文件系统扫描
元数据漂移      标签命名不一致、分类错误            规则校验 + 小模型审查
冗余积累        同一信息被多次记录                  相似度检测
知识失效        过去的决策已被推翻但未标记          大模型定期审查
```

### 保鲜策略

```
数据类型          保鲜周期     保鲜方式
──────────────────────────────────────────────────────
任务状态          实时         采集层事件驱动更新
日历/会议         每小时       与飞书日历对比
文档摘要          每日         对比文档最后编辑时间
人/项目元数据     每周         与飞书通讯录/项目对比
知识库内容        每月         大模型审查
双向链接完整性    每日         文件系统扫描
标签一致性        每周         规则 + 小模型检查
审计数据          每月聚合     详细数据 → 统计摘要
```

### FreshnessEngine

FreshnessEngine 是存储层信息保鲜的执行引擎。它不独立调度，而是作为 [定时任务系统](scheduler.md) 中 S-01 到 S-09 的具体实现。

```
FreshnessEngine
  │
  ├── 调度器
  │     ├── 由 Scheduler 统一调度（不独立持有调度器）
  │     └── 避开用户活跃时段（默认凌晨执行）
  │
  ├── SyncTask（同步类）── 对应 Scheduler 任务
  │     ├── 任务状态同步       → S-01
  │     ├── 日历同步           → C-01（采集层）
  │     ├── 文档变更检测       → S-02
  │     └── 通讯录同步         → C-03（采集层）
  │
  ├── IntegrityTask（完整性类）── 对应 Scheduler 任务
  │     ├── 双向链接检查       → S-03
  │     ├── 引用完整性         → S-03
  │     ├── 三层一致性         → S-06
  │     └── 孤儿文件检测       → S-03
  │
  ├── QualityTask（质量类）── 对应 Scheduler 任务
  │     ├── 重复检测           → S-04
  │     ├── 标签规范化         → S-05
  │     ├── 摘要新鲜度         → S-02
  │     └── 知识过时检测       → S-08
  │
  ├── CleanupTask（清理类）── 对应 Scheduler 任务
  │     ├── 审计数据聚合       → S-07
  │     ├── 向量 DB 增量同步   → S-09
  │     └── 临时文件清理       → S-09
  │
  └── 保鲜报告
        └── 每次执行后生成报告
```

### 用户介入机制

```
自动修复（静默执行）：
  - 任务状态同步
  - 断链标记（不删除，只标记）
  - 标签拼写变体合并

用户确认（推送通知）：
  - 疑似重复记录合并
  - 知识过时标记确认
  - 未归类文档的分类

用户主动触发（设置中操作）：
  - 向量 DB 全量重建
  - 历史数据重新处理
  - 存储层一致性全量校验
```
