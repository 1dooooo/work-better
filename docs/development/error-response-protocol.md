# 错误响应协议

> 当用户明确指出错误时，AI 必须遵守此协议。

## 核心原则

**禁止直接改代码。必须先反思开发交付流程。**

## 触发条件

以下情况必须触发此协议：

- 用户说"这里有 bug"
- 用户说"这个功能不对"
- 用户说"测试为什么没有发现"
- 用户明确指出任何代码或功能错误

## 响应流程

```
用户指出错误
    ↓
Step 1: 暂停（不改代码）
    ↓
Step 2: 反思开发流程
    - 这个错误是如何产生的？
    - 测试为什么没有发现？
    - 代码审查为什么没有发现？
    - 验收测试为什么没有发现？
    - 开发流程中哪个环节有漏洞？
    ↓
Step 3: 找出根因
    - 是测试覆盖不足？
    - 是测试验证的是 mock 而非真实行为？
    - 是代码审查清单有遗漏？
    - 是完成标准不严格？
    - 是代码复用缺乏强制约束？
    ↓
Step 4: 修复流程
    - 更新测试要求
    - 更新代码审查清单
    - 更新完成标准
    - 更新开发规范
    ↓
Step 5: 修复代码
    - 在流程修复后，再修复当前问题
```

## 示例

### 错误响应

```
用户："这里的逻辑不对"
AI："好的，我来修复"（直接改代码）
```

### 正确响应

```
用户："这里的逻辑不对"
AI："让我先分析一下这个错误是如何产生的..."
     1. 检查测试覆盖情况
     2. 检查代码审查清单
     3. 检查完成标准
     4. 找出流程漏洞
     5. 提出流程修复方案
     6. 修复流程
     7. 修复代码
```

## 为什么重要

1. **治标不治本**：直接改代码只能解决当前问题，不能防止类似问题再次出现
2. **流程漏洞**：错误的产生往往意味着开发流程有漏洞，需要从流程层面解决
3. **持续改进**：通过反思和修复流程，可以不断提升开发质量
4. **避免重复**：记录流程漏洞和修复方案，可以防止同样的问题重复出现

## 反思记录模板

每次反思都必须记录到 `docs/decisions/` 目录：

```markdown
# ADR-XXX: [错误描述]

## 问题
用户指出 [具体错误]

## 反思
这个错误是如何通过开发流程的？
- 测试覆盖：[分析]
- 代码审查：[分析]
- 完成标准：[分析]

## 根因
[流程中的漏洞]

## 流程修复
[更新了哪些规范/清单/标准]

## 代码修复
[修复了什么代码]

## 防止再次发生
[如何防止类似问题]
```
